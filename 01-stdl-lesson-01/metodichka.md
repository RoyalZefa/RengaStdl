# Урок 1: первый шаблон стиля в Renga STDL

В этом уроке мы разберем два файла:

- `parameters.json`
- `main.lua`

Цель урока очень простая: понять, как из параметров получается объект в Renga.

У вас уже есть рабочий пример `TutorialBox`, значит теперь можно изучать его как учебный.

## Что вы уже сделали

Вы уже прошли важный путь:

1. скачали `RengaSTDLSDK`
2. собрали `RST` файл
3. импортировали его в категории Renga
4. увидели стиль `TutorialBox`

Теперь наша задача не просто "заставить работать", а понять, как это устроено.

## Как мыслить про STDL

В STDL есть две главные части:

1. `parameters.json` описывает параметры стиля
2. `main.lua` строит геометрию по этим параметрам

Проще говоря:

- `JSON` отвечает за то, что пользователь может настраивать
- `Lua` отвечает за то, как по этим настройкам будет выглядеть объект

Можно запомнить так:

- `parameters.json` = "что можно вводить"
- `main.lua` = "что из этого строится"

## Часть 1. Разбираем parameters.json

Откройте файл `01-stdl-lesson-01/parameters.json`.

В нем сейчас такая структура:

```json
{
  "metadata": { ... },
  "styleParameters": [ ... ],
  "ports": []
}
```

Разберем по блокам.

### 1. Блок metadata

Вот он:

```json
"metadata": {
  "defaultName": "TutorialBox",
  "description": "A minimal STDL lesson project for Renga Style Template Scripting.",
  "version": "1.0.0",
  "author": "Alex"
}
```

Зачем нужен этот блок:

- `defaultName` - имя стиля, которое вы видите в Renga
- `description` - описание шаблона
- `version` - версия вашего шаблона
- `author` - автор

Практика:

1. поменяйте `defaultName` с `TutorialBox` на `MyFirstBox`
2. соберите `RST` заново
3. импортируйте его в Renga
4. посмотрите, что название стиля стало другим

Важно:

изменение `metadata` не меняет геометрию. Оно меняет "паспорт" шаблона.

### 2. Блок styleParameters

Это самая важная часть JSON.

В нашем примере есть одна группа:

```json
{
  "name": "Dimensions",
  "text": "Dimensions",
  "params": [ ... ]
}
```

Здесь:

- `name` - техническое имя группы
- `text` - название группы, которое показывается пользователю

Внутри группы лежат параметры:

- `Width`
- `Depth`
- `Height`

Пример одного параметра:

```json
{
  "name": "Width",
  "text": "Width",
  "type": "Length",
  "default": 1000,
  "min": 100,
  "max": 10000
}
```

Что означают поля:

- `name` - как к параметру обращаться из Lua
- `text` - как подписать его в интерфейсе Renga
- `type: "Length"` - это размер
- `default` - значение по умолчанию
- `min` и `max` - границы, чтобы пользователь не ввел что-то нереальное

Самая важная мысль:

именно `name` связывает `JSON` и `Lua`.

Если в `JSON` параметр называется `Width`, то в `Lua` вы должны обращаться именно к `Width`.

Практика:

1. поменяйте `default` у `Width` с `1000` на `1500`
2. соберите файл заново
3. импортируйте в Renga
4. посмотрите, что объект стал шире по умолчанию

### 3. Блок ports

Сейчас у нас:

```json
"ports": []
```

Это означает:

у нас пока нет портов.

Порты нужны, когда объект должен подключаться к инженерной системе:

- труба
- воздуховод
- кабель и так далее

Для первого урока это правильно. Мы начинаем без портов, чтобы не усложнять пример.

## Мини-итог по parameters.json

После этого блока вы должны чувствовать следующее:

1. `metadata` - общая информация о стиле
2. `styleParameters` - параметры, которые пользователь меняет
3. `ports` - точки подключения, пока пустые

Если сказать очень коротко:

`parameters.json` описывает интерфейс вашего будущего объекта.

## Часть 2. Разбираем main.lua

Теперь откройте файл `01-stdl-lesson-01/main.lua`.

Здесь описано, как строится объект.

Разберем сверху вниз.

### 1. Читаем параметры

Вот первые строки:

```lua
local parameters = Style.GetParameterValues()

local width = parameters.Dimensions.Width
local depth = parameters.Dimensions.Depth
local height = parameters.Dimensions.Height
```

Что здесь происходит:

- `Style.GetParameterValues()` забирает все значения из `parameters.json`
- дальше мы достаем конкретно `Width`, `Depth`, `Height`

Обратите внимание на связь:

- `Dimensions` - это имя группы из `JSON`
- `Width`, `Depth`, `Height` - имена параметров из `JSON`

Это главная логика STDL:

сначала вы описываете параметры в `JSON`, потом используете их в `Lua`.

### 2. Вывод в лог

Дальше есть строка:

```lua
print("TutorialBox:", width, depth, height)
```

Она не строит геометрию.

Она нужна для отладки.

Когда шаблон работает, вы можете смотреть в лог и понять:

- какие значения реально пришли
- что скрипт вообще выполнился

Это очень полезно, когда объект "не такой", как вы ожидали.

### 3. Функция makeTopViewSymbol

Вот эта часть:

```lua
local function makeTopViewSymbol()
    local geometrySet = GeometrySet2D()
    local rectangle = CreateRectangle2D(Point2D(0, 0), 0, width, depth)
    geometrySet:AddCurve(rectangle)
    return geometrySet
end
```

Это функция, которая строит 2D-прямоугольник.

Зачем он нужен:

- чтобы у объекта было условное обозначение
- чтобы он понятно выглядел на виде сверху

По сути:

- создали 2D-набор геометрии
- создали прямоугольник
- добавили его в набор
- вернули результат

Важно понять:

это еще не 3D-модель.
Это только 2D-графика.

### 4. Создаем детальную 3D-геометрию

Вот главная часть:

```lua
local detailedGeometry = ModelGeometry()
detailedGeometry:AddSolid(CreateBlock(width, depth, height))
Style.SetDetailedGeometry(detailedGeometry)
```

Разбор:

- `ModelGeometry()` создает контейнер для геометрии
- `CreateBlock(width, depth, height)` создает параллелепипед
- `AddSolid(...)` добавляет его в модель
- `Style.SetDetailedGeometry(...)` говорит Renga: вот основная геометрия объекта

Именно эта часть делает из настроек реальный 3D-объект.

Если упростить до одного предложения:

мы берем `width`, `depth`, `height` и строим из них коробку.

### 5. Создаем symbolic geometry

Дальше:

```lua
local symbolicGeometry = ModelGeometry()
local topViewPlacement = Placement3D(
    Point3D(0, 0, height),
    Vector3D(0, 0, 1),
    Vector3D(1, 0, 0)
)
symbolicGeometry:AddGeometrySet2D(makeTopViewSymbol(), topViewPlacement)
Style.SetSymbolicGeometry(symbolicGeometry)
```

Здесь уже интереснее.

Мы:

1. создаем контейнер для символической геометрии
2. задаем расположение этой 2D-геометрии в 3D-пространстве
3. добавляем наш 2D-прямоугольник
4. передаем это в Renga как symbolic geometry

Почему `Point3D(0, 0, height)`:

потому что мы кладем 2D-обозначение на верхнюю плоскость объекта.

То есть символ рисуется на высоте `height`.

## Мини-итог по main.lua

Вот что должно остаться в голове:

1. читаем параметры из `JSON`
2. строим 3D-форму
3. строим 2D-обозначение
4. передаем оба вида геометрии в Renga

Если очень просто:

`main.lua` - это инструкция для Renga, как из параметров сделать объект.

## Как повторить это руками в Renga

Теперь практическая часть.

Сделайте по шагам.

### Упражнение 1. Меняем имя стиля

В `parameters.json`:

- измените `defaultName`

Потом:

1. соберите `RST`
2. импортируйте в Renga
3. проверьте, что стиль получил новое имя

Чему вы научитесь:

понять, за что отвечает `metadata`.

### Упражнение 2. Меняем размеры по умолчанию

В `parameters.json`:

- поменяйте `default` у `Width`
- поменяйте `default` у `Height`

Потом:

1. соберите `RST`
2. импортируйте в Renga
3. создайте объект
4. сравните его с прошлой версией

Чему вы научитесь:

понять, как параметры влияют на модель.

### Упражнение 3. Меняем форму 2D-обозначения

В `main.lua` внутри `makeTopViewSymbol()` попробуйте:

- оставить прямоугольник как есть
- или позже добавить внутренний прямоугольник

Пример идеи:

```lua
local outerRect = CreateRectangle2D(Point2D(0, 0), 0, width, depth)
local innerRect = CreateRectangle2D(Point2D(0, 0), 0, width - 100, depth - 100)
geometrySet:AddCurve(outerRect)
geometrySet:AddCurve(innerRect)
```

Чему вы научитесь:

понять разницу между 3D-геометрией и 2D-символом.

### Упражнение 4. Меняем 3D-геометрию

Самый простой эксперимент:

временно замените:

```lua
CreateBlock(width, depth, height)
```

на блок с другими коэффициентами, например:

```lua
CreateBlock(width, depth, height / 2)
```

Тогда вы увидите:

- параметры остались те же
- но геометрия стала ниже

Чему вы научитесь:

понять, что `Lua` может свободно интерпретировать параметры, а не просто "копировать" их.

## Самая важная связка

Запомните эту одну линию мысли:

1. параметр объявлен в `parameters.json`
2. параметр прочитан в `main.lua`
3. параметр использован при построении геометрии
4. результат видно в Renga

Если вы понимаете эту цепочку, значит основа STDL у вас уже есть.

## Что делать, если ничего не меняется

Если вы правите файл, но в Renga не видите изменений, проверьте по порядку:

1. точно ли вы заново собрали `RST`
2. точно ли импортировали новую версию в проект
3. точно ли создаете объект из нового стиля
4. нет ли ошибок в логе Renga

Лог для `print()` и ошибок:

```text
%LOCALAPPDATA%\Renga Software\Renga\AecApp.log
```
